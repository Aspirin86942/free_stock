# 东方财富掘金实盘执行系统规划书（第一期）

## 文档定位

本文定义第一期最小闭环，并以当前已确认的实现口径为准。第一期目标是在东方财富掘金仿真账户中跑通“持仓识别 -> 规则判断 -> 自动卖出 -> 查询收口 -> 日志审计”。

当前项目统一采用**查询驱动**架构：

- 下单后先拿同步提交结果
- 再通过 `query_order_status()` 确认委托终态
- 如订单出现成交，再通过 `query_execution_reports()` 确认成交明细
- 查询结果先整理成内部事件，再由状态聚合逻辑或后续状态机消费

## 一、目标与边界

### 1.1 目标

- 连接东方财富掘金仿真账户
- 读取账户资金、全部持仓和相关行情
- 自动识别全部可卖持仓
- 按固定止盈止损规则触发自动卖出
- 下单后通过查询确认委托状态与成交结果
- 防止同一标的重复卖出
- 留存可审计日志

### 1.2 范围

- 账户、持仓、行情、委托、查单、查成交全部统一走东方财富掘金官方接口
- 开发语言固定为 Python 3.10+
- 运行形态为单进程常驻程序
- 只做自动卖出，不做自动买入
- 持仓默认由人工先建，系统不负责开仓
- 第一版按单账户、单策略、统一参数运行

### 1.3 非范围

- 前端
- 回测主链路
- 数据库主链路
- 对外 Web API
- 多账户
- 多策略组合
- 动态脚本加载
- 第二期和远期路线的完整展开

### 1.4 成功标准

在东方财富掘金仿真环境中，系统能够自动识别账户当前全部可卖持仓，并对满足固定止盈止损条件的标的执行自动卖出；每个标的都具备独立状态管理、重复卖单防护和可审计日志；下单后的最终状态能够被查询确认。

## 二、技术基线

| 维度 | 第一阶段方案 |
| --- | --- |
| 交易环境 | 东方财富掘金仿真账户 |
| 接口来源 | 东方财富掘金官方接口 |
| 开发语言 | Python 3.10+ |
| 运行方式 | 单进程常驻程序 |
| 策略范围 | 固定止盈止损 |
| 交易方向 | 仅自动卖出 |
| 收口方式 | 查询驱动 |
| 观测手段 | 本地日志 + 查单结果 + 查成交结果 |
| 部署目标 | 先跑通最小闭环，不追求平台化 |

## 三、系统架构

### 3.1 四层架构

| 分层 | 职责 |
| --- | --- |
| 基础设施层 | 配置、启动停止、调度、交易时段控制、日志、异常处理 |
| 数据接入层 | 掘金账户连接、资金读取、持仓读取、行情读取、委托调用、查单、查成交 |
| 核心决策层 | 逐标的状态管理、止盈止损判断、卖出许可判断 |
| 交易执行层 | 卖单执行、订单跟踪、成交收口、防重复卖单 |

### 3.2 架构约束

- 所有外部交易与行情调用统一经过数据接入层
- 核心决策层只负责“该不该卖”，不直接发单
- 交易执行层只负责“怎么卖”和“卖后状态怎么收口”
- 状态必须按标的隔离，禁止用单一全局布尔值代替
- 第一阶段不把供应商异步通知作为主流程输入
- 查询结果必须先收敛成内部统一对象，再进入状态聚合或后续状态机

## 四、核心运行链路

1. 程序启动，加载配置并初始化日志。
2. 基础设施层判断当前运行环境和交易时段。
3. 数据接入层连接掘金仿真账户并确认账户可用。
4. 数据接入层读取账户资金和当前全部持仓。
5. 核心决策层筛选全部可卖持仓，并为每个标的建立或更新状态。
6. 数据接入层获取相关标的最新行情。
7. 核心决策层依据持仓成本、当前价格、止盈比例和止损比例判断是否触发卖出。
8. 核心决策层检查该标的是否已有未完成卖单，以及当前是否允许发单。
9. 交易执行层发起卖单，并记录触发原因、数量、价格和提交结果。
10. 交易执行层按固定节奏查询订单状态和成交明细。
11. 查询结果先转换成内部事件，再更新内存态聚合结果；后续接入数据库时，事件将驱动状态表。
12. 程序继续轮询全部持仓，直到停止。

## 五、里程碑

| 里程碑 | 目标 | 完成定义 |
| --- | --- | --- |
| M0 | 环境与账户连通 | 能完成鉴权，并读取账户资金、全部持仓和持仓标的行情 |
| M1 | 双向手工验证层 | 能手动提交一笔 `buy/sell` 委托，并通过查单与查成交确认最终状态 |
| M2 | 核心决策与状态管理 | 多标的状态管理、止盈止损判断、卖出许可判断可运行 |
| M3 | 自动卖出执行闭环 | 满足条件的标的能自动卖出，并完成查询驱动的状态收口 |
| M4 | 测试、日志与稳定运行 | 关键测试可执行，日志齐全，程序可连续稳定运行 |

## 六、测试与质量保障

### 6.1 测试范围

- 单元测试：止盈判断、止损判断、交易时段判断、防重复卖单、逐标的状态隔离
- 集成测试：持仓读取到决策输出、决策输出到交易执行、查单查成交到状态收口
- 仿真冒烟：账户连通、持仓读取、触发卖出、查单确认、异常落日志

### 6.2 最低质量门槛

- 无持仓时不发单
- 非交易时段不发单
- 同一标的不重复发单
- 接口失败和查询异常必须落日志
- 核心判断口径可通过测试复现

## 七、风险与运行约束

- 仿真环境只能验证链路正确性，不等于真实成交结果
- 非交易时段允许启动和读取状态，但禁止发起新卖单
- 同一标的若已有未完成卖单，必须禁止再次主动发单
- 多个标的必须独立维护状态，不能互相覆盖
- 行情、账户、委托、查单、查成交异常必须进入明确状态并落日志
- 第一阶段状态只落在内存和日志；后续扩展数据库后，再由内部事件驱动状态表并定时与供应商核对
- 账户无持仓时系统只能空转，不允许自行开仓

## 八、结论

第一期只追求一件事：把东方财富掘金仿真环境下的自动卖出最小闭环跑通。当前项目基线不是“等外部通知”，而是“提交结果 + 查询确认 + 内部事件聚合”。只要 `M0 -> M4` 能按上述口径落地，这一版规划书就达到了项目要求。
