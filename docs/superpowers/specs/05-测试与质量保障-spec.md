# 测试与质量保障 Spec

> 本文件已按 2026-04-10 的查询驱动架构修订。第一期验证基线是“提交结果 + 主动查询 + 内部事件聚合”，不是依赖供应商 callback。

## 1. 文档目标

本文定义第一期实盘执行系统的测试范围、验证方式、质量门槛和交付证据。

测试与质量保障不是附属条目，而是系统能否进入仿真运行的硬门槛。没有这一层约束，即使代码能够启动，也无法证明它不会误卖、重复卖或在错误状态下继续运行。

## 2. 本节定位

本节覆盖整个第一期系统，但重点服务以下目标：

- 保证关键判断可复现
- 保证查询驱动主链路可验证
- 保证错误不会静默穿透
- 保证输出结果可审计、可回放、可定位

## 3. 测试范围与非范围

### 3.1 测试范围

- 基础设施层的配置、调度、交易时段判断
- 数据接入层的对象转换、结构化错误、主动查询契约
- `M1` 双向手工验证层的查询驱动闭环
- `M2` 核心决策与状态管理 dry-run
- `M3` 自动卖出执行层的查询驱动收口和防重复卖单
- 仿真环境下的最小闭环冒烟验证

### 3.2 非范围

- 回测收益验证
- 高频性能压测
- 多账户并发验证
- 第二阶段平台化能力验证

## 4. 与里程碑关系

| 里程碑 | 测试重点 |
| --- | --- |
| `M0` | 启动、配置、账户连通、资金/持仓/行情可读 |
| `M1` | `buy/sell` 双向手工验证，提交后可通过查单和查成交确认状态 |
| `M2` | 多标的决策态、止盈止损判断、`should_sell/can_submit_sell` 分离、墓碑态输出 |
| `M3` | 自动卖出执行、防重复卖单、查询驱动的执行态收口 |
| `M4` | 日志完整、错误可审计、程序可稳定连续运行 |

## 5. 测试分层策略

### 5.1 单元测试

目标：验证纯逻辑、纯转换和纯聚合的正确性。

至少覆盖：

- 配置校验
- 交易时段判断
- 止盈止损判断
- `should_sell` 与 `can_submit_sell` 的分离
- `trigger_reason` 与 `block_reason` 的分离
- `volume > 0` 且 `available_volume = 0` 的处理
- `quote_missing`、墓碑态等边界路径
- 查询快照到内部事件的标准化
- 内部事件到聚合报告或状态更新的聚合逻辑

### 5.2 集成测试

目标：验证模块之间的关键链路没有断点。

至少覆盖：

- 配置加载到运行上下文
- 持仓和行情进入 `M2` 后能产出稳定的 `DecisionResult`
- `M1` 链路：`submit_order -> query_order_status -> query_execution_reports -> 内部事件 -> TradeReport`
- `M2` 链路：`positions -> quotes -> state sync -> decision evaluate -> M2RoundReport`
- `M3` 链路：`decision facts + query-driven execution facts -> 执行态更新`
- 查询异常、缺行情和拒单场景能被结构化表达

### 5.3 仿真冒烟验证

目标：验证真实掘金仿真链路至少跑通一次。

至少验证：

- 能连接掘金仿真账户
- 能读取账户资金、全部持仓和相关行情
- `M1` 能手工提交一笔 `buy` 或 `sell` 并通过查询确认结果
- `M2` 能输出至少一轮 `m2_round_summary`
- 若 `M3` 已实现，触发卖出后能通过查询确认终态和成交明细

明确要求：

- 不把供应商 callback 作为冒烟通过前提
- 若真实环境缺少某一步验证条件，必须记录未执行原因

### 5.4 运行期观察验证

目标：验证常驻程序在连续运行场景下没有出现明显失控。

至少关注：

- 日志是否按轮输出
- 接口失败和脏数据是否落日志
- `M2` 的变化标的是否可审计
- `M2` 的 `quote_missing` 和墓碑态是否有明确观察载体
- `M3` 中已有未完成卖单时是否被正确拦截

## 6. 关键测试对象

### 6.1 基础设施层

- 缺失必填配置时是否阻止启动
- 日志目录初始化失败时是否阻止启动
- 轮询间隔非法时是否阻止启动
- 交易时段判断是否正确区分 `pre_open`、`trading`、`post_close`、`closed_day`

### 6.2 数据接入层

- 账户快照字段是否完整
- 持仓快照字段是否完整
- 行情快照字段是否完整
- 委托提交结果是否包含明确状态和标识
- `query_order_status()` 返回的状态快照是否完整
- `query_execution_reports()` 返回的成交快照是否完整
- 查询结果是否能稳定整理为内部事件

### 6.3 核心决策层

- 当前价格达到止盈阈值时是否触发
- 当前价格跌破止损阈值时是否触发
- 达到阈值但非交易时段时是否只触发、不允许提交
- 达到阈值但 `available_volume = 0` 时是否显式输出 `temporarily_not_closable`
- 缺行情时是否显式输出 `quote_missing`
- 标的消失时是否进入一轮墓碑态
- 多标的状态是否互不污染

### 6.4 交易执行层

- 接收到卖出许可后是否正确生成委托请求
- 提交前是否再次检查未完成卖单
- 委托提交失败时是否进入明确失败路径
- 部分成交后是否继续保持可跟踪状态
- 已成交完成后是否正确结束收口
- 执行态是否基于查询结果而不是 callback 假设推进

## 7. 测试数据与环境约束

### 7.1 本地测试环境

- Python 3.10+
- 默认优先使用 `conda run -n stock_analysis ...`
- 非必要场景不依赖真实柜台
- 测试桩和假网关应优先覆盖外部接口分支

### 7.2 仿真验证环境

- 使用东方财富掘金仿真账户
- 使用专用测试配置，不与未来实盘配置混用
- 冒烟测试前必须确认测试账户状态可控
- 若需要人为准备持仓，应在测试记录中注明准备方式

### 7.3 测试数据要求

持仓数据至少覆盖：

- 无持仓
- 单标的持仓
- 多标的持仓
- `volume > 0` 且 `available_volume = 0`
- 标的在下一轮消失

委托与成交数据至少覆盖：

- `pending_new` 或等价非终态
- `submitted`
- `partially_filled`
- `filled`
- `cancelled`
- `rejected`

错误数据至少覆盖：

- 鉴权失败
- 行情缺失
- 查单失败
- 查成交失败
- 柜台返回无法识别的状态

## 8. 质量门槛

第一期最低质量门槛如下：

- 无持仓不发单
- 非交易时段不发单
- 同一标的不重复发单
- `M2` 对暂不可平仓和缺行情场景有显式输出
- 接口失败和查询异常必须落日志
- 关键判断和关键状态演进可通过测试复现
- 关键链路至少完成一次仿真冒烟

## 9. 交付证据

每个里程碑完成时，至少保留以下证据：

- 测试命令
- 测试结果摘要
- 冒烟日志文件或截图
- 失败样例及处理记录

若某项验证因外部环境无法执行，必须明确写出未执行原因，不得默认为“已通过”。

## 10. 推荐验证命令

第一期建议至少保留以下命令约定：

```powershell
conda run -n stock_analysis pytest
conda run -n stock_analysis pytest tests/unit
conda run -n stock_analysis pytest tests/integration
conda run -n stock_analysis python main.py --config config/sim_account.yaml
conda run -n stock_analysis python main.py --config config/sim_account.yaml --mode m1 --side sell --symbol SHSE.600839 --volume 100 --price-type market --timeout-seconds 60
conda run -n stock_analysis python main.py --config config/sim_account.yaml --mode m2 --once
```

如果后续项目目录调整，命令可以变，但必须保留“单元测试、集成测试、仿真冒烟”三类入口。

## 11. 本节完成定义

测试与质量保障可视为完成，至少需要满足以下条件：

- 已定义统一测试分层口径
- `M1/M2/M3` 都有明确验证目标
- 查询驱动主链路具备可执行测试清单
- 质量门槛可被执行和复查
- 每个里程碑都知道要交什么验证证据
