# 基础设施层 Spec

## 1. 文档目标

本文定义第一期实盘执行系统的基础设施层边界、职责、输入输出、运行约束和验收标准。

基础设施层不是“杂项收纳层”，而是第一期最先落地的底座。它必须先解决下面几个问题：

- 程序怎么启动
- 配置怎么读取
- 日志怎么记录
- 交易时段怎么控制
- 轮询怎么调度
- 异常怎么暴露
- M0 怎么验证

如果这一层没有先定清楚，后续数据接入层和交易执行层即使能写，也会处于不可运维、不可验证的状态。

## 2. 本层定位

基础设施层负责系统运行期的公共能力，不负责业务判断，也不直接承担掘金接口调用。

本层固定包含以下职责：

- 配置管理
- 程序启动与停止控制
- 调度与轮询节奏控制
- 交易时段控制
- 日志与审计基础能力
- 异常处理与退出策略

## 3. 范围与非范围

### 3.1 本层范围

- 加载本地配置文件并完成最小校验
- 初始化运行目录和日志目录
- 控制程序启动、运行、停止
- 提供固定轮询节奏
- 判断当前是否允许进入交易判断和交易执行
- 提供统一日志输出格式
- 定义最小测试与验证命令

### 3.2 不在本层范围

- 掘金账户登录与鉴权
- 资金、持仓、行情读取
- 止盈止损判断
- 卖单执行
- 委托与成交回报业务处理

这些能力分别属于数据接入层、核心决策层和交易执行层。

## 4. 与里程碑关系

基础设施层直接支撑以下里程碑：

- `M0 环境与账户连通`
- `M4 测试、日志与稳定运行`

其中，`M0` 的落地前提并不是“已经会交易”，而是：

- 程序能够正常启动
- 配置能够正常读取
- 日志能够正常输出
- 运行时段控制可用
- 能为数据接入层提供稳定入口

## 5. 输入、输出与依赖

### 5.1 输入

- 本地配置文件
- 启动命令参数
- 当前本地时间
- 运行环境信息

### 5.2 输出

- 已校验的运行配置
- 统一日志输出
- 当前运行状态
- 允许或禁止交易判断
- 轮询触发信号

### 5.3 依赖

- Python 3.10+
- 标准库日志能力
- 标准库时间与调度能力
- 本地文件系统

第一期不依赖数据库、消息队列、HTTP 服务或第三方调度平台。

## 6. 配置契约

第一期基础设施层至少要识别以下配置项：

| 配置项 | 必填 | 说明 |
| --- | --- | --- |
| `account_id` | 是 | 掘金交易账户标识 |
| `token` | 是 | 掘金鉴权令牌 |
| `strategy_name` | 是 | 当前策略实例名称，用于日志标识 |
| `poll_interval_seconds` | 是 | 主轮询间隔，单位秒 |
| `take_profit_ratio` | 是 | 止盈比例 |
| `stop_loss_ratio` | 是 | 止损比例 |
| `trade_session_start` | 是 | 允许进入交易判断的开始时间 |
| `trade_session_end` | 是 | 允许进入交易判断的结束时间 |
| `log_dir` | 是 | 日志目录 |
| `timezone` | 否 | 默认 `Asia/Shanghai` |

### 6.1 配置校验规则

- 缺少必填项时，程序禁止启动。
- `poll_interval_seconds` 必须大于零。
- `take_profit_ratio` 和 `stop_loss_ratio` 必须是正数。
- 交易开始时间必须早于交易结束时间。
- `log_dir` 不存在时可自动创建，但创建失败必须中止启动。

## 7. 启动与停止流程

### 7.1 启动流程

程序启动时必须按以下顺序执行：

1. 读取配置文件
2. 校验关键配置
3. 初始化日志目录与日志对象
4. 输出启动日志
5. 判断当前运行环境是否合法
6. 进入主循环，等待数据接入层工作

### 7.2 停止流程

程序停止时必须：

1. 记录停止原因
2. 输出最终运行状态
3. 确保日志句柄正常关闭

若因异常停止，必须把异常堆栈和最后一次运行上下文写入错误日志。

## 8. 调度与交易时段控制

### 8.1 调度要求

第一期不引入复杂调度系统，只保留单进程固定轮询。

调度要求如下：

- 主循环按固定秒级间隔执行
- 单次轮询超时或异常不能导致后续轮询彻底失效
- 日志中必须能看到每一轮触发时间

### 8.2 交易时段判断

基础设施层必须提供统一的交易时段判断能力，供核心决策层和交易执行层复用。

第一期至少区分以下状态：

- `pre_open`：未开盘，不允许发单
- `trading`：交易时段，允许判断与发单
- `post_close`：收盘后，不允许发单
- `closed_day`：非交易日，不允许发单

第一期可以先按固定时间窗口控制，不要求先接交易日历服务，但必须保留后续替换空间。

### 8.3 并发控制

第一期采用**单线程顺序处理模型**，避免并发带来的竞态条件和状态不一致问题。

#### 8.3.1 执行模型

主循环按以下顺序执行：

1. 处理所有待处理的回报事件（从回调队列中取出）
2. 读取持仓和行情
3. 执行决策判断
4. 提交卖单（如果触发）
5. 等待下一轮

#### 8.3.2 约束

- 回报事件通过回调函数放入队列，由主循环统一处理
- 判断和发单在同一个线程中顺序执行
- 禁止在回调函数中直接更新状态或执行业务逻辑
- 禁止使用多线程并发处理业务逻辑

#### 8.3.3 第二期扩展

如需引入并发，必须：

- 对状态管理器的读写操作加锁
- 使用原子操作检查并更新状态
- 明确定义事件处理的优先级和互斥规则

## 9. 日志与审计基础能力

### 9.1 日志分类

基础设施层至少要支持三类日志：

- 运行日志
- 交易日志
- 错误日志

### 9.2 日志最小字段

所有日志至少包含：

- 时间
- 日志级别
- 策略实例名
- 模块名
- 事件类型
- 消息正文

错误日志额外要求包含：

- 异常类型
- 错误原因
- 关键上下文

### 9.3 审计要求

基础设施层必须保证：

- 启动、停止、异常退出都有日志
- 每轮调度都有日志
- 每次策略触发和下单动作能够被后续模块写入审计日志

### 9.4 关键业务事件日志格式

关键业务事件必须使用**结构化日志**，格式为：

```
{timestamp} {level} {strategy_name} {event_type} {key1}={value1} {key2}={value2} ...
```

示例：

```
2026-04-08 10:00:00 INFO gmtrade-live-m0 heartbeat round=123 positions=2 open_orders=0
2026-04-08 10:00:01 INFO gmtrade-live-m0 decision_triggered symbol=SHSE.600036 trigger_type=take_profit cost_price=10.00 current_price=10.52 threshold=10.50 volume=100
2026-04-08 10:00:02 INFO gmtrade-live-m0 order_submitted symbol=SHSE.600036 order_id=ORDER_123 volume=100 price_type=market
2026-04-08 10:00:05 INFO gmtrade-live-m0 order_filled symbol=SHSE.600036 order_id=ORDER_123 filled_volume=100 avg_price=10.51
```

关键事件类型：

- `heartbeat` - 心跳日志（每轮循环）
- `decision_triggered` - 触发卖出信号
- `decision_blocked` - 卖出被阻断
- `order_submitted` - 委托已提交
- `order_failed` - 委托失败
- `order_filled` - 委托成交
- `order_cancelled` - 委托撤单
- `order_partial_filled` - 部分成交
- `state_change` - 状态变化
- `error` - 错误事件

这种格式便于：

1. 人工阅读
2. 脚本解析（grep、awk）
3. 导入日志分析系统（ELK、Splunk）

## 10. 异常处理策略

第一期禁止静默失败。

基础设施层异常处理规则如下：

- 配置错误：直接阻止启动
- 日志初始化失败：直接阻止启动
- 单轮运行异常：记录错误后进入下一轮
- 致命异常：记录错误后退出程序

第一期不引入自动重启守护，但必须给后续接入外部守护留空间。

### 10.1 错误分类与处理策略

| 错误类型 | 是否重试 | 重试策略 | 失败后处理 |
|---------|---------|---------|-----------|
| 配置错误 | 否 | - | 阻止启动 |
| 鉴权失败 | 否 | - | 阻止启动 |
| 日志初始化失败 | 否 | - | 阻止启动 |
| 网络超时 | 是 | 下一轮重试 | 记录日志，跳过本轮 |
| 行情读取失败 | 是 | 下一轮重试 | 记录日志，该标的不进入判断 |
| 持仓读取失败 | 是 | 下一轮重试 | 记录日志，跳过本轮判断 |
| 委托提交失败 | 否 | - | 进入 failed 状态，记录日志 |
| 回报解析失败 | 否 | - | 记录原始载荷，进入异常状态 |
| 状态持久化失败 | 是 | 立即重试 1 次 | 记录日志，不影响主流程 |

第一期不引入指数退避、熔断等复杂策略。

测试与质量保障总则由 [05-测试与质量保障-spec.md](D:\Program_python\free_stock\docs\superpowers\specs\05-测试与质量保障-spec.md) 单独定义；本节只保留基础设施层自己的最低验证要求。

### 11.1 单元测试

至少覆盖：

- 配置校验
- 交易时段判断
- 轮询参数校验
- 启动前置检查

### 11.2 集成测试

至少覆盖：

- 配置加载到运行上下文的链路
- 日志初始化成功路径

### 11.3 冒烟验证

## 12. 本层完成定义

基础设施层可视为完成，至少需要满足以下条件：

- 程序能稳定启动和停止
- 配置有明确契约和校验
- 日志目录和日志输出可用
- 交易时段控制可复用
- 固定轮询机制可运行
- 最小测试与冒烟验证可执行

## 13. 对下游层的交付物

基础设施层完成后，应为后续分层提供以下稳定基线：

- 已校验的配置对象
- 统一日志接口
- 运行状态上下文
- 当前交易时段判断结果
- 固定调度入口

这 5 项是数据接入层能够开始落地的前提。

## 14. 监控与告警

### 14.1 心跳日志

每轮循环必须输出心跳日志：

```
2026-04-08 10:00:00 INFO gmtrade-live-m0 heartbeat round=123 positions=2 open_orders=0
```

如果心跳日志超过 2 个轮询周期未更新，说明程序可能卡死。

### 14.2 健康检查文件

每轮循环更新健康检查文件：

```json
{
  "last_update": "2026-04-08T10:00:00",
  "status": "running",
  "round": 123,
  "positions": 2,
  "open_orders": 0
}
```

文件路径：`logs/health.json`

外部监控脚本可以定期检查此文件的 `last_update` 时间戳。

### 14.3 告警（第二期）

第二期可以考虑：

- 邮件告警
- 企业微信/钉钉告警
- 短信告警

## 15. 配置安全性

### 15.1 敏感信息存储

第一期使用环境变量存储敏感信息（账户 ID、Token、数据库密码等）。

配置文件示例：

```yaml
account_id: ${GM_ACCOUNT_ID}
token: ${GM_TOKEN}
db_password: ${MYSQL_PASSWORD}
```

启动前设置环境变量：

```powershell
$env:GM_ACCOUNT_ID = Read-Host "GM_ACCOUNT_ID"
$env:GM_TOKEN = Read-Host "GM_TOKEN" -AsSecureString | ConvertFrom-SecureString
```

### 15.2 安全风险

环境变量方案存在以下风险：

- 环境变量可能被其他进程读取
- PowerShell 历史记录可能泄露敏感信息
- 进程崩溃时环境变量可能被转储

### 15.3 缓解措施

1. 使用 `Read-Host` 而不是直接 `$env:VAR = "value"`
2. 定期清理 PowerShell 历史：`Clear-History`
3. 不在脚本中硬编码敏感信息
4. 配置文件不提交到 git（通过 `.gitignore` 排除）

### 15.4 第二期改进

第二期可以考虑：

- 使用 Windows Credential Manager
- 使用配置加密（如 `cryptography` 库）
- 使用密钥管理服务（如 Azure Key Vault、HashiCorp Vault）
