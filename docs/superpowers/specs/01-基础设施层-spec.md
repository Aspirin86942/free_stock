# 基础设施层 Spec

> 本文件已按最新查询驱动架构修订。

## 1. 文档目标

本文定义第一期实盘执行系统的基础设施层边界、职责、输入输出、运行约束和验收标准。

基础设施层先解决四件事：

- 程序怎么启动
- 配置怎么读取
- 轮询怎么调度
- 日志怎么留下

如果这一层不先收紧，后续数据接入、决策和执行就会失去统一入口。

## 2. 本层定位

基础设施层负责系统运行期的公共能力，不负责业务判断，也不直接承担掘金接口调用。

本层固定包含以下职责：

- 配置管理
- 程序启动与停止控制
- 调度与轮询节奏控制
- 交易时段控制
- 日志与审计基础能力
- 异常处理与退出策略

## 3. 范围与非范围

### 3.1 本层范围

- 加载本地配置文件并完成最小校验
- 初始化运行目录和日志目录
- 控制程序启动、运行、停止
- 提供固定轮询节奏
- 判断当前是否允许进入交易判断和交易执行
- 提供统一日志输出格式
- 定义最小测试与验证命令

### 3.2 不在本层范围

- 账户鉴权
- 资金、持仓、行情读取
- 止盈止损判断
- 卖单执行
- 订单状态与成交收口细节

这些能力分别属于数据接入层、核心决策层和交易执行层。

## 4. 输入、输出与依赖

### 4.1 输入

- 本地配置文件
- 启动命令参数
- 当前本地时间
- 运行环境信息

### 4.2 输出

- 已校验的运行配置
- 统一日志输出
- 当前运行状态
- 允许或禁止交易判断
- 固定轮询触发信号

### 4.3 依赖

- Python 3.10+
- 标准库日志能力
- 标准库时间能力
- 本地文件系统

第一期不依赖数据库、消息队列、HTTP 服务或第三方调度平台。

## 5. 配置契约

第一期基础设施层至少要识别以下配置项：

| 配置项 | 必填 | 说明 |
| --- | --- | --- |
| `account_id` | 是 | 掘金交易账户标识 |
| `token` | 是 | 掘金鉴权令牌 |
| `strategy_name` | 是 | 当前策略实例名称，用于日志标识 |
| `poll_interval_seconds` | 是 | 主轮询间隔，单位秒 |
| `take_profit_ratio` | 是 | 止盈比例 |
| `stop_loss_ratio` | 是 | 止损比例 |
| `trade_session_start` | 是 | 允许进入交易判断的开始时间 |
| `trade_session_end` | 是 | 允许进入交易判断的结束时间 |
| `log_dir` | 是 | 日志目录 |
| `timezone` | 否 | 默认 `Asia/Shanghai` |

### 5.1 校验规则

- 缺少必填项时，程序禁止启动
- `poll_interval_seconds` 必须大于零
- `take_profit_ratio` 和 `stop_loss_ratio` 必须是正数
- 交易开始时间必须早于交易结束时间
- `log_dir` 不存在时可自动创建，但创建失败必须中止启动

## 6. 启动与停止流程

### 6.1 启动流程

程序启动时必须按以下顺序执行：

1. 读取配置文件
2. 校验关键配置
3. 初始化日志目录与日志对象
4. 输出启动日志
5. 判断当前运行环境是否合法
6. 进入主循环

### 6.2 停止流程

程序停止时必须：

1. 记录停止原因
2. 输出最终运行状态
3. 确保日志句柄正常关闭

若因异常停止，必须把异常堆栈和最后一次运行上下文写入错误日志。

## 7. 调度与交易时段控制

### 7.1 调度要求

第一期不引入复杂调度系统，只保留单进程固定轮询。

调度要求如下：

- 主循环按固定秒级间隔执行
- 单次轮询超时或异常不能导致后续轮询彻底失效
- 日志中必须能看到每一轮触发时间

### 7.2 交易时段判断

基础设施层必须提供统一的交易时段判断能力，供核心决策层和交易执行层复用。

第一期至少区分以下状态：

- `pre_open`
- `trading`
- `post_close`
- `closed_day`

第一期可以先按固定时间窗口控制，不要求先接交易日历服务，但必须保留后续替换空间。

### 7.3 执行模型

第一期采用**单线程顺序处理模型**，主循环按以下顺序执行：

1. 对未完成订单执行一轮状态对账
2. 读取账户、持仓和行情
3. 执行决策判断
4. 提交新卖单
5. 等待下一轮

### 7.4 约束

- 不以供应商异步通知作为主流程输入
- 查询结果必须先转成内部统一对象，再交给状态聚合逻辑
- 判断和发单在同一个线程中顺序执行
- 禁止使用多线程并发处理业务逻辑

## 8. 日志与审计基础能力

### 8.1 日志分类

基础设施层至少要支持三类日志：

- 运行日志
- 交易日志
- 错误日志

### 8.2 日志最小字段

所有日志至少包含：

- 时间
- 日志级别
- 策略实例名
- 模块名
- 事件类型
- 消息正文

错误日志额外要求包含：

- 异常类型
- 错误原因
- 关键上下文

### 8.3 审计要求

基础设施层必须保证：

- 启动、停止、异常退出都有日志
- 每轮调度都有日志
- 每次策略触发和下单动作都能被后续模块继续补充审计信息

### 8.4 结构化日志格式

关键业务事件必须使用结构化日志，格式为：

```text
{timestamp} {level} {strategy_name} {event_type} {key1}={value1} {key2}={value2} ...
```

示例：

```text
2026-04-08 10:00:00 INFO gmtrade-live heartbeat round=123 positions=2 open_orders=0
2026-04-08 10:00:01 INFO gmtrade-live decision_triggered symbol=SHSE.600036 trigger_type=take_profit cost_price=10.00 current_price=10.52 threshold=10.50 volume=100
2026-04-08 10:00:02 INFO gmtrade-live order_submitted symbol=SHSE.600036 cl_ord_id=ORDER_123 volume=100 price_type=market
2026-04-08 10:00:05 INFO gmtrade-live order_status_reconciled symbol=SHSE.600036 status=filled filled_volume=100
```

## 9. 验收标准

- 能稳定加载配置并启动
- 能输出结构化日志
- 能按固定节奏执行主循环
- 能正确区分交易时段状态
- 能在异常场景下保留可审计日志
