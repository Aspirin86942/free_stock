# 东方财富掘金第一期最小实盘闭环规划书修订设计

## 1. 目标

本设计文档用于指导整体重写 [量化交易系统规划书.md](D:\Program_python\free_stock\Proposal\量化交易系统规划书.md)。

本次不是在原有研究型量化系统规划书上做局部修补，而是将其整体改写为面向第一期最小实盘闭环的规划书。原文档中的研究、回测、前端展示、多数据源采集和远期路线内容均不再保留为第一期主线。

## 2. 新文档定位

重写后的 Proposal 文档定位为：

`东方财富掘金实盘执行系统规划书（第一期）`

文档目标聚焦于一个最小可运行、可验证、可审计的仿真实盘闭环，而不是研究型量化平台。

## 3. 已确认范围

### 3.1 执行环境

- 交易环境固定为东方财富掘金仿真账户。
- 行情、账户、持仓、委托、回报全部统一走东方财富掘金接口。
- 第一期开发表达语言固定为 Python。

### 3.2 交易范围

- 第一版不限制单标的，改为自动识别账户当前全部持仓。
- 系统对所有“有可卖数量的持仓标的”执行统一止盈止损判断。
- 第一版只做自动卖出，不做自动买入。
- 持仓前提由人工建仓提供，系统不负责开仓。

### 3.3 策略范围

- 第一版只内置固定止盈止损规则。
- 策略触发依据为持仓成本、当前价格、止盈比例和止损比例。
- 不引入均线、突破、多因子或外部信号驱动逻辑。

### 3.4 明确不做

- 前端页面
- 回测功能
- 自动买入
- 多账户支持
- 对外 Web API
- 数据库主链路
- 多策略组合
- 任意 Python 策略上传
- 第二期或远期路线说明

## 5. 建议的新章节结构

### 5.1 项目定位与边界

说明第一期只做什么、不做什么，并把成功标准写清楚。

### 5.2 总体技术方案

说明第一期采用 Python + 东方财富掘金官方接口 + 单进程策略程序，不再混用新浪、AkShare 或其他数据源。

### 5.3 系统架构设计

采用实盘执行系统分层，而不是研究平台分层。核心分层固定为：

- 基础设施层
- 数据接入层
- 核心决策层
- 交易执行层

### 5.4 核心运行链路

主链路必须描述为：

1. 程序启动
2. 连接掘金仿真账户
3. 读取账户资金与全部持仓
4. 过滤出可卖持仓
5. 逐标的获取价格并做止盈止损判断
6. 检查该标的是否已有未完成卖单
7. 触发卖出委托
8. 接收委托与成交回报
9. 更新逐标的状态
10. 记录日志并继续运行

### 5.5 模块设计说明

必须至少覆盖：

- 掘金接入模块
- 持仓与状态模块
- 卖出策略模块
- 订单执行模块
- 调度与日志模块

### 5.6 测试与质量保障

测试与质量保障必须作为单独章节存在，不能只附属于基础设施层的一句说明。

本章节至少要写清：

- 单元测试：止盈止损判断、交易时段判断、防重复卖单
- 集成测试：持仓读取、状态更新、信号触发到委托执行之间的关键链路
- 仿真冒烟验证：掘金仿真账户下的最小自动卖出闭环
- 验证边界：无持仓、有持仓但未触发、触发卖出、已有未完成卖单、回报异常

### 5.7 里程碑设计

里程碑需改成围绕最小实盘闭环推进：

- M0：环境与账户连通
- M1：数据接入跑通
- M2：核心决策与状态管理
- M3：自动卖出执行闭环
- M4：测试、日志与稳定运行

### 5.8 风险与运行约束

必须重点写清：

- 仿真与真实交易差异
- 重复卖单风险
- 多标的状态串扰风险
- 行情、账户、委托回报异常风险
- 交易时段控制
- 自动买入不在第一期范围

## 6. 关键设计口径

### 6.1 运行方式

- 第一阶段采用单进程常驻策略程序。
- 优先使用掘金提供的实时事件与定时能力完成轮询与触发。
- 不引入前端、不建设 HTTP 服务、不建设数据库主链路。

### 6.2 四层架构职责

#### 6.2.1 基础设施层

负责：

- 配置管理
- 程序启动与停止
- 调度与交易时段控制
- 日志与审计
- 测试与质量保障
- 异常处理与重试预留

#### 6.2.2 数据接入层

负责：

- 掘金账户登录与鉴权
- 账户资金读取
- 持仓读取
- 行情读取
- 委托下单
- 委托与成交回报接收

这一层只负责和掘金交互，不做业务判断。

#### 6.2.3 核心决策层

负责：

- 逐标的状态管理
- 止盈止损判断
- 是否允许卖出
- 是否已有未完成委托
- 是否满足交易时段约束

这一层输出应当明确表达：

- 是否卖出
- 触发原因
- 当前状态
- 是否允许进入执行层

#### 6.2.4 交易执行层

负责：

- 接收核心决策层输出
- 发起卖单
- 跟踪委托状态
- 更新逐标的成交结果
- 保证同一标的不重复卖出

### 6.3 多标的处理方式

- 系统启动后自动读取账户当前全部持仓。
- 仅对可卖数量大于零的标的进行判断。
- 每个标的必须维护独立状态，避免状态串扰。

### 6.4 防重复卖出要求

- 同一标的若已有未完成卖单，禁止再次发起主动卖单。
- 不同标的的卖出状态必须彼此独立。
- 程序重启后应至少根据账户持仓和未完成委托恢复基本状态判断。

### 6.5 日志与审计要求

第一期即使没有数据库，也必须保留最小审计能力。

日志至少包含：

- 时间
- 标的代码
- 当前价格
- 持仓数量
- 持仓成本
- 止盈阈值
- 止损阈值
- 触发原因
- 委托结果
- 成交状态
- 错误原因

## 7. 成功标准

重写后的 Proposal 文档必须能准确表达以下成功定义：

在东方财富掘金仿真环境中，系统能够自动识别账户当前全部可卖持仓，并对满足固定止盈止损条件的标的执行自动卖出；整个过程中，每个标的都具备独立状态管理、重复卖出防护和可审计日志。
