# 数据接入层 Spec

> 本文件已按最新查询驱动架构修订。

## 1. 文档目标

本文定义第一期实盘执行系统的数据接入层边界、职责、内部契约、异常口径和验收标准。

第一期的数据接入层只做一件事：把东方财富掘金官方接口转换成系统内部可稳定消费的快照、提交结果和查询结果。它不负责业务判断，也不负责策略状态流转。

## 2. 本层定位

数据接入层是系统唯一的外部接口入口，统一负责：

- 掘金账户连接与鉴权
- 账户资金读取
- 持仓读取
- 行情读取
- 卖出委托提交
- 委托状态查询
- 成交明细查询

本层必须把掘金 SDK 或接口的原始结构收敛为系统内部统一对象，避免上层直接依赖第三方字段。

## 3. 范围与非范围

### 3.1 本层范围

- 初始化掘金运行环境
- 完成账户连通和鉴权
- 拉取账户资产、可用资金和持仓
- 拉取持仓标的行情
- 提交卖出委托
- 查询委托状态
- 查询成交明细
- 输出统一错误对象和原始响应审计信息

### 3.2 不在本层范围

- 止盈止损判断
- 交易时段判断
- 卖出许可判断
- 防重复卖单规则本身
- 状态聚合与状态机推进

这些能力分别属于基础设施层、核心决策层和交易执行层。

## 4. 与里程碑关系

数据接入层直接支撑以下里程碑：

- `M0 环境与账户连通`
- `M1 手动交易查询收口`
- `M3 自动卖出执行闭环`

其中：

- `M0` 关注账户、资金、持仓、行情是否可读
- `M1` 关注委托提交、查单、查成交是否跑通
- `M3` 关注本层能否稳定支撑上层形成完整卖出闭环

## 5. 输入、输出与依赖

### 5.1 输入

- 已校验的运行配置
- 掘金鉴权信息
- 账户标识
- 目标标的列表
- 卖出委托请求
- 内部委托号

### 5.2 输出

- 账户快照
- 持仓快照列表
- 行情快照列表
- 委托提交结果
- 委托状态快照
- 成交明细快照列表
- 统一错误对象

### 5.3 依赖

- Python 3.10+
- 东方财富掘金官方 Python 接口
- 本地网络环境
- 基础设施层提供的配置、日志和运行上下文

## 6. 内部能力拆分

### 6.1 账户连接与鉴权

职责：

- 初始化掘金运行环境
- 加载账户和鉴权配置
- 验证账户是否可用

要求：

- 鉴权失败必须明确阻止后续读取和下单
- 启动阶段必须输出连通结果日志
- 不允许在连通失败时伪造空账户对象继续运行

### 6.2 账户与持仓读取

职责：

- 读取账户总资产、可用资金、持仓市值
- 读取当前全部持仓
- 提取每个标的的可卖数量和持仓成本

要求：

- 读取结果必须保留原始时间戳
- 持仓为空时返回空列表，而不是异常
- 持仓字段缺失时必须返回明确错误

### 6.3 行情读取

职责：

- 针对当前持仓标的读取最新行情
- 向上层提供统一价格视图

要求：

- 第一版至少提供最新价和时间戳
- 行情读取失败时必须显式返回失败，不得沿用旧值冒充新值
- 上层若需要批量读取，应由本层统一封装批量入口

### 6.4 委托提交

职责：

- 接收交易执行层生成的卖出请求
- 调用掘金官方接口发起卖单
- 返回委托是否被受理及对应标识

要求：

- 第一版只支持卖出方向
- 本层只负责调用成功与否，不负责“该不该下单”
- 委托提交结果必须保留原始响应摘要，便于审计

### 6.5 委托状态查询

职责：

- 按内部委托号查询订单当前状态
- 统一映射状态码、数量和错误原因
- 输出 `OrderStatusSnapshot`

要求：

- 结果必须包含 `cl_ord_id`、`broker_order_id`、`symbol`、`status`、`filled_volume`、`remaining_volume`、`event_time`
- 无法识别的状态码必须保留原始上下文并落错误日志
- 查不到订单时返回 `None`，由上层决定是否继续轮询

### 6.6 成交明细查询

职责：

- 按内部委托号查询成交明细
- 统一映射成交量、均价和时间
- 输出 `tuple[OrderExecutionSnapshot, ...]`

要求：

- 查询结果必须只包含当前内部委托号对应的记录
- 单条成交价格必须统一为 `Decimal`
- 没有成交记录时返回空元组，而不是异常

## 7. 查询驱动契约

第一期统一采用**查询驱动**：

1. 下单后先拿同步提交结果
2. 主流程按固定节奏查单
3. 订单进入成交相关状态后再查成交
4. 查询结果先转成内部事件，再由上层聚合或后续状态机消费

### 7.1 Gateway 协议

当前交易网关至少暴露以下方法：

```python
class TradeGateway(Protocol):
    def connect(self, config: AppConfig) -> None: ...
    def get_cash(self, account_id: str) -> CashSnapshot: ...
    def get_positions(self, account_id: str) -> list[PositionSnapshot]: ...
    def submit_order(self, request: OrderRequest) -> OrderSubmitResult: ...
    def query_order_status(self, cl_ord_id: str, symbol: str) -> OrderStatusSnapshot | None: ...
    def query_execution_reports(self, cl_ord_id: str) -> tuple[OrderExecutionSnapshot, ...]: ...
```

### 7.2 内部事件口径

本层对外暴露的是快照对象；上层在消费时应把快照进一步整理成内部事件，再驱动状态聚合。第一期内部事件至少要能表达：

- 一次查单结果
- 一次查成交结果
- 一笔订单当前聚合状态

## 8. 数据契约

### 8.1 账户快照

| 字段 | 说明 |
| --- | --- |
| `account_id` | 账户标识 |
| `available_cash` | 可用资金 |
| `market_value` | 持仓市值 |
| `total_asset` | 总资产 |
| `update_time` | 快照时间 |

### 8.2 持仓快照

| 字段 | 说明 |
| --- | --- |
| `symbol` | 标的代码 |
| `exchange` | 交易所标识 |
| `volume` | 总持仓数量 |
| `available_volume` | 当前可卖数量 |
| `cost_price` | 持仓成本 |
| `last_update_time` | 快照时间 |

### 8.3 行情快照

| 字段 | 说明 |
| --- | --- |
| `symbol` | 标的代码 |
| `last_price` | 最新价 |
| `quote_time` | 行情时间 |
| `source` | 行情来源 |

### 8.4 委托提交结果

| 字段 | 说明 |
| --- | --- |
| `accepted` | 是否被接口受理 |
| `cl_ord_id` | 内部委托号 |
| `broker_order_id` | 柜台委托号 |
| `symbol` | 标的代码 |
| `message` | 接口返回说明 |
| `raw_status` | 原始状态摘要 |
| `event_time` | 响应时间 |

### 8.5 委托状态快照

| 字段 | 说明 |
| --- | --- |
| `cl_ord_id` | 内部委托号 |
| `broker_order_id` | 柜台委托号 |
| `symbol` | 标的代码 |
| `status` | 当前状态 |
| `filled_volume` | 已成交数量 |
| `remaining_volume` | 剩余数量 |
| `rejection_reason` | 拒单或失败原因 |
| `event_time` | 查询结果时间 |

### 8.6 成交快照

| 字段 | 说明 |
| --- | --- |
| `cl_ord_id` | 内部委托号 |
| `broker_order_id` | 柜台委托号 |
| `symbol` | 标的代码 |
| `filled_volume` | 本条成交数量 |
| `avg_price` | 成交均价 |
| `event_time` | 查询结果时间 |

## 9. 错误口径

- 字段缺失必须抛出结构化错误
- 查询接口无结果时，返回空对象或 `None`，不得伪造成功
- 柜台拒绝、余额不足、仓位不足等业务错误必须保留原始原因
- 时间字段缺失时可以使用当前时间补齐，但必须在实现中统一处理

## 10. 验收标准

- 能完成账户连通
- 能稳定读取账户、持仓、行情
- 能提交一笔卖单
- 能通过查单确认订单状态
- 能在有成交时通过查成交确认数量和均价
- 所有数据都被转换成统一内部对象
