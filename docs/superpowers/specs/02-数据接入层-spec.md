# 数据接入层 Spec

## 1. 文档目标

本文定义第一期实盘执行系统的数据接入层边界、职责、内部契约、异常口径和验收标准。

第一期的数据接入层只做一件事：把东方财富掘金官方接口转换成系统内部可稳定消费的数据和事件。它不负责业务判断，也不负责策略状态流转。

## 2. 本层定位

数据接入层是系统唯一的外部接口入口，统一负责：

- 掘金账户连接与鉴权
- 账户资金读取
- 持仓读取
- 行情读取
- 卖出委托提交
- 委托与成交回报接收

本层必须把掘金 SDK 或接口的原始结构收敛为系统内部统一对象，避免上层直接依赖第三方字段。

## 3. 范围与非范围

### 3.1 本层范围

- 初始化掘金运行环境
- 完成账户连通和鉴权
- 拉取账户资产、可用资金和持仓
- 拉取或订阅持仓标的行情
- 提交卖出委托
- 接收委托状态和成交回报
- 输出统一错误对象和原始响应审计信息

### 3.2 不在本层范围

- 止盈止损判断
- 交易时段判断
- 卖出许可判断
- 防重复卖单规则本身
- 逐标的业务状态机

这些能力分别属于基础设施层、核心决策层和交易执行层。

## 4. 与里程碑关系

数据接入层直接支撑以下里程碑：

- `M0 环境与账户连通`
- `M1 数据接入跑通`
- `M3 自动卖出执行闭环`

其中：

- `M0` 关注账户、资金、持仓、行情是否可读
- `M1` 关注委托提交和回报接收是否跑通
- `M3` 关注本层能否稳定支撑上层形成完整卖出闭环

## 5. 输入、输出与依赖

### 5.1 输入

- 已校验的运行配置
- 掘金鉴权信息
- 账户标识
- 目标标的列表
- 卖出委托请求

### 5.2 输出

- 账户快照
- 持仓快照列表
- 行情快照列表
- 委托提交结果
- 委托事件和成交事件
- 统一错误对象

### 5.3 依赖

- Python 3.10+
- 东方财富掘金官方 Python 接口
- 本地网络环境
- 基础设施层提供的配置、日志和运行上下文

第一期不混用 AkShare、新浪、腾讯或其他第三方行情源。

## 6. 内部能力拆分

### 6.1 账户连接与鉴权

职责：

- 初始化掘金运行环境
- 加载账户和鉴权配置
- 验证账户是否可用

要求：

- 鉴权失败必须明确阻止后续读取和下单
- 启动阶段必须输出连通结果日志
- 不允许在连通失败时伪造空账户对象继续运行

### 6.2 账户与持仓读取

职责：

- 读取账户总资产、可用资金、持仓市值
- 读取当前全部持仓
- 提取每个标的的可卖数量和持仓成本

要求：

- 读取结果必须保留原始时间戳
- 持仓为空时返回空列表，而不是异常
- 持仓字段缺失时必须返回明确错误

### 6.3 行情读取

职责：

- 针对当前持仓标的读取最新行情
- 向上层提供统一价格视图

要求：

- 第一版至少提供最新价和时间戳
- 行情读取失败时必须显式返回失败，不得沿用旧值冒充新值
- 上层若需要批量读取，应由本层统一封装批量入口

### 6.4 委托提交

职责：

- 接收交易执行层生成的卖出请求
- 调用掘金官方接口发起卖单
- 返回委托是否被受理及对应标识

要求：

- 第一版只支持卖出方向
- 本层只负责调用成功与否，不负责“该不该下单”
- 委托提交结果必须保留原始响应摘要，便于审计

### 6.5 委托与成交回报接收

职责：

- 接收委托状态变化
- 接收成交回报
- 把回报转换为统一内部事件

要求：

- 回报事件必须包含订单标识、标的代码、状态、数量、时间
- 无法识别的回报状态必须落错误日志
- 回报不能直接改写上层业务状态，只能输出事件

## 7. 数据契约

### 7.1 账户快照

| 字段 | 说明 |
| --- | --- |
| `account_id` | 账户标识 |
| `available_cash` | 可用资金 |
| `market_value` | 持仓市值 |
| `total_asset` | 总资产 |
| `update_time` | 快照时间 |

### 7.2 持仓快照

| 字段 | 说明 |
| --- | --- |
| `symbol` | 标的代码 |
| `exchange` | 交易所标识 |
| `volume` | 总持仓数量 |
| `available_volume` | 当前可卖数量 |
| `cost_price` | 持仓成本 |
| `last_update_time` | 快照时间 |

### 7.3 行情快照

| 字段 | 说明 |
| --- | --- |
| `symbol` | 标的代码 |
| `last_price` | 最新价 |
| `quote_time` | 行情时间 |
| `source` | 固定为 `gmtrade` |

### 7.4 卖出委托请求

| 字段 | 说明 |
| --- | --- |
| `symbol` | 标的代码 |
| `volume` | 卖出数量 |
| `side` | 固定为 `sell` |
| `price_type` | 市价或限价类型 |
| `price` | 限价单价格，可为空 |

### 7.5 委托提交结果

| 字段 | 说明 |
| --- | --- |
| `accepted` | 是否被接口受理 |
| `order_id` | 委托标识 |
| `symbol` | 标的代码 |
| `message` | 接口返回说明 |
| `raw_status` | 原始状态摘要 |
| `event_time` | 响应时间 |

### 7.6 回报事件

| 字段 | 说明 |
| --- | --- |
| `order_id` | 委托标识 |
| `symbol` | 标的代码 |
| `event_type` | `order` 或 `trade` |
| `status` | 当前状态 |
| `filled_volume` | 已成交数量 |
| `remaining_volume` | 剩余数量 |
| `avg_price` | 成交均价 |
| `event_time` | 回报时间 |

## 8. 错误处理策略

第一期禁止把接口问题隐藏为业务空结果。

错误处理口径如下：

- 鉴权失败：直接中止启动
- 账户读取失败：返回显式错误，保留错误上下文
- 持仓读取失败：返回显式错误，当前轮不进入后续判断
- 行情读取失败：返回显式错误，当前标的不进入卖出判断
- 委托提交失败：返回失败结果，由交易执行层决定如何收口
- 回报解析失败：记录错误日志，并保留原始载荷摘要

第一期不引入复杂重试编排，默认由下一轮调度重新拉起读取链路。

## 9. 日志与审计要求

本层至少记录以下事件：

- 账户连通成功或失败
- 账户读取成功或失败
- 持仓读取成功或失败
- 行情读取成功或失败
- 卖出委托提交请求和结果
- 委托回报和成交回报

关键日志至少包含：

- 账户标识
- 标的代码
- 接口动作
- 返回状态
- 时间
- 错误原因

## 10. 对上游和下游层的接口要求

### 10.1 对基础设施层的要求

- 提供已校验配置
- 提供统一日志对象
- 提供运行上下文和时区

### 10.2 对核心决策层的交付物

- 统一账户快照
- 统一持仓快照
- 统一行情快照
- 显式错误对象

### 10.3 对交易执行层的交付物

- 委托提交能力
- 委托提交结果
- 委托与成交回报事件流

## 11. 测试要求

虽然测试章节单独成册，但本层必须至少满足以下最小测试口径：

- 鉴权失败时能否正确阻止启动
- 账户对象能否正确转换为内部快照
- 持仓对象能否正确转换为内部快照
- 行情对象能否正确转换为内部快照
- 委托提交结果能否正确转换为内部结果
- 回报事件能否正确转换为统一事件对象

## 12. 本层完成定义

数据接入层可视为完成，至少需要满足以下条件：

- 掘金账户连通可验证
- 账户、持仓、行情可稳定读取
- 卖出委托可提交
- 委托与成交回报可接收
- 所有返回结构已统一收敛
- 关键错误不会被吞掉
